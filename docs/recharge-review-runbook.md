# 充值人工审核操作手册

## 前置条件

1. 已执行 `supabase/migrations/20260704_recharge_review_flow.sql`。
2. 已执行创建 `complete_account_recharge` 的充值入账 RPC migration。
3. 已创建私有 Storage Bucket：`payment-proofs`。
4. 手工充值渠道已在 `payment_channels` 中启用并标记为已配置。

## 用户侧流程

1. 用户进入账户充值页。
2. 选择金额和已启用渠道。
3. 系统创建充值单，状态为 `waiting_payment`。
4. 用户提交交易流水号、付款时间、付款账号摘要、备注和最多 3 个凭证文件。
5. 系统将充值状态改为 `submitted`。

## 管理员审核流程

1. 进入后台支付管理。
2. 打开对应充值记录详情。
3. 查看用户摘要、申请金额、渠道、凭证、交易流水号、历史充值数量和审核事件。
4. 点击“开始审核”，状态变为 `reviewing`。
5. 核对到账信息后选择：
   - 审核通过：必须填写原因，并二次确认。
   - 驳回：必须填写原因。
   - 要求补充凭证：必须填写原因，用户可重新提交。
   - 取消充值：必须填写原因。
6. 审核通过后系统调用 `complete_account_recharge` 入账。
7. 入账成功后状态为 `succeeded`；失败则保持 `approved` 并记录错误，可点击“重新处理入账”。

## 幂等与失败恢复

- 同一充值单成功后不能再次审核入账。
- 重复点击审核通过由状态机和 RPC 共同保护。
- 入账失败不会标记为 `succeeded`。
- 失败重试必须再次走 `retry_credit`，不能直接修改余额。

## 安全规则

- 不允许管理员自由修改用户余额。
- 不允许普通用户查看其他用户凭证。
- 不允许凭证提交直接增加余额。
- 不允许未配置渠道生成假二维码或假地址。
- 审核理由、状态变化和入账失败都会留痕。

## 常见问题

### 渠道显示暂未开放

检查 `payment_channels.enabled` 和 `payment_channels.configured`，并确认公开配置中的审核模式或 Provider 配置完整。

### 凭证上传失败

检查文件类型、文件大小、Bucket 是否存在，以及 Service Role 是否仅在服务端可用。

### 审核通过后入账失败

检查 `complete_account_recharge` 是否存在，充值金额、币种和交易流水号是否满足 RPC 校验。

### 重复交易流水号被拒绝

这是预期保护。同一 Provider 交易号不能对应多个充值单。
