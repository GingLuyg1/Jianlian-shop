# 负载与容量测试方案

## 测试环境要求

- 仅允许在自有测试环境执行，不得对生产域名压测。
- 使用测试 Supabase 项目、测试商品、测试账号和测试订单。
- 支付 Provider 未接入时只能验证未配置降级，不生成真实支付成功。
- 测试前记录 commit、环境变量名称、数据库 migration 执行状态和 PM2 实例数。

## 接口分类

- 公开读取：首页、商品列表、商品搜索、商品详情。
- 认证写入：订单创建、充值创建、退款申请、支付会话创建。
- 高频查询：支付状态查询、订单列表、后台控制台统计。
- 管理写入：商品保存、SKU 保存、库存导入、媒体上传、余额调整。
- 内部任务：自动发货重试、支付对账、数据一致性巡检。

## 建议并发级别

- 商品列表和商品详情：10、25、50 并发逐级验证。
- 登录、注册、忘记密码：低速率验证限流，不做高压冲击。
- 订单创建和支付会话：1、3、5 并发验证幂等与库存，不做真实付款。
- 后台统计：1、3、5 并发，观察数据库查询时间。
- 库存导入：单管理员单任务验证，不并发导入同一商品。

## 请求速率与持续时间

- 公开读取：每档 3 分钟，观察 P95、P99 和错误率。
- 资金和库存接口：短时 burst 验证限流，重点检查重复订单、重复入账和重复发货。
- 管理接口：按管理员维度验证 429，不把管理员接口视为无限流量。

## 基线指标

- 首页、商品列表、商品详情 P95 建议低于 800ms。
- 订单创建、支付会话创建 P95 建议低于 1500ms。
- 后台统计 P95 建议低于 2000ms，失败时模块化降级。
- 错误率目标低于 1%，限流 429 不计入系统错误。
- 数据库连接数、CPU、内存必须记录峰值。

## 停止条件

- 出现重复订单、重复充值入账、重复发货或库存超卖。
- 出现 5xx 持续超过 1 分钟。
- 数据库连接接近上限或响应明显雪崩。
- 支付、订单、库存日志出现敏感信息。
- 任一测试影响真实用户或生产资源。

## 数据清理原则

- 不删除真实历史数据。
- 测试商品、测试订单和测试库存必须带明确测试标记。
- 数字库存测试内容使用无效卡密，并在报告中记录批次。
- 不清理支付、余额和订单审计链路，使用状态标记区分测试数据。
