# 数据一致性巡检验收报告

## 巡检规则清单

规则集中定义在 `lib/consistency/rules.ts`，当前覆盖：

- OP-001：已支付订单缺少成功支付记录，P0
- OP-002：支付成功但订单仍待支付，P0
- OP-003：订单金额与支付金额不一致，P0
- OP-004：订单币种与支付币种不一致，P1
- OP-005：渠道交易号关联多个订单，P0
- OP-006：单个订单存在多个成功支付记录，P0
- OP-007：取消订单仍存在有效支付会话，P1
- RB-001：充值成功但缺少入账流水，P0
- RB-002：失败充值生成余额流水，P0
- RB-003：同一充值重复入账，P0
- RB-004：余额流水引用不存在的充值，P1
- RB-005：充值金额与入账金额不一致，P1
- RB-006：余额前后值不连续，P1
- RF-001：退款金额超过订单实付金额，P0
- RF-002：退款成功但缺少余额退款流水，P1
- RF-003：退款币种与订单币种不一致，P1
- ID-001：同一库存被重复预留，P0
- ID-002：同一库存被重复交付，P0
- ID-003：已交付库存缺少交付记录，P1
- ID-004：交付记录引用可用库存，P0
- ID-005：预留库存缺少有效订单，P1
- ID-006：数字库存跨 SKU 关联，P0
- ID-007：订单数量与交付数量不一致，P1

## 订单支付检查结果

已实现只读检查：

- 订单已支付但无成功支付记录。
- 支付成功但订单仍待支付。
- 订单金额与支付金额不一致。
- 订单币种与支付币种不一致。
- 渠道交易号关联多个订单。
- 单订单多条成功支付记录。
- 取消订单仍有有效支付会话。

检查只输出订单号、支付单号、会话号和摘要，不输出完整回调、密钥或签名。

## 充值余额检查结果

已实现只读检查：

- 充值成功无入账流水。
- 失败或关闭充值存在入账流水。
- 同一充值重复入账。
- 充值流水引用不存在的充值单。
- 充值金额与入账流水金额不一致。
- 同一用户余额流水前后值不连续。

巡检不会通过流水求和覆盖当前余额，也不会增加或扣减余额。

## 退款检查结果

已实现只读检查：

- 退款累计超过订单实付金额。
- 余额退款成功但缺少余额退款流水。
- 退款币种与订单币种不一致。

Provider 未配置时不会推断外部退款成功。

## 库存交付检查结果

已实现只读检查：

- 同一库存重复预留。
- 同一库存重复交付。
- delivered 库存缺少交付记录。
- 交付记录引用 available 库存。
- reserved 库存缺少有效订单。
- 多 SKU 商品库存与订单项 SKU 不一致。
- 订单项交付数量与购买数量不一致。

输出只包含库存 ID、商品 ID、SKU ID、订单 ID 和交付 ID 摘要，不输出完整数字库存内容。

## 巡检记录结构

新增 migration：`supabase/migrations/20260630_data_consistency_scan.sql`

新增表：

- `public.data_consistency_runs`
- `public.data_consistency_issues`

异常使用 `fingerprint` 聚合，相同异常会更新出现次数、首次发现和最后发现时间。

## 后台巡检页面

新增页面：`/admin/system/data-consistency`

功能：

- 查看最近巡检时间、状态、规则数、异常数、P0/P1、已解决和待处理数量。
- 按严重程度、规则编号和处理状态筛选。
- 查看异常安全详情和修复建议。
- 标记处理中、已解决、已忽略，并要求填写备注。
- 处理操作写入管理员审计日志。

页面不提供任意 SQL、余额调整、支付状态修改、库存释放或重分配按钮。

## 异常聚合方式

- 指纹来源：规则编号、实体类型、实体 ID 和脱敏关联实体摘要。
- 同一指纹重复出现时更新 `occurrences` 和 `last_seen_at`。
- 已解决异常再次出现会重新打开为 `open`。

## 修复建议方式

每条规则包含固定建议，例如：

- 运行支付对账。
- 检查订单支付 RPC。
- 检查充值入账 RPC。
- 检查退款记录。
- 释放无效预留。
- 人工补发。
- 检查 Migration 和 RLS。

建议只提供处理方向，不提供危险数据修改按钮。

## 权限与数据脱敏

- 后台页面和 API 仅允许超级管理员邮箱访问。
- 内部接口需要 `DATA_CONSISTENCY_SCAN_SECRET`。
- 普通用户不能读取巡检记录。
- 巡检不保存支付密钥、完整回调原文、完整数字库存、卡密、账号密码。
- 处理状态变更写入 `admin_audit_logs`。

## 发现的问题

- 项目此前没有统一的数据一致性巡检记录表。
- 项目此前没有订单、支付、充值、余额、退款、库存和交付的一体化只读巡检入口。

## 已修复的问题

- 新增集中巡检规则定义。
- 新增只读巡检服务。
- 新增巡检记录 migration。
- 新增后台巡检页面和内部巡检接口。
- 新增运行手册和验收报告。

## 仍存在的问题

- 需要人工在 Supabase 执行 migration 后，后台页面才能持久化巡检记录。
- 定时任务需要人工配置，不由应用自动写入 Crontab/PM2/Nginx。
- Provider 未接入时只能检查本站支付数据，不能自动确认外部渠道状态。
- 真实数据巡检结果需要连接生产或测试 Supabase 后由超级管理员执行确认。

## 需要执行的 Migration

1. 确认业务表、支付表、审计表 migration 已执行。
2. 执行 `supabase/migrations/20260630_data_consistency_scan.sql`。
