# USDT-BEP20 人工复核手册

## 欠额转余额的管理员流程

1. 仅超级管理员进入后台支付管理的“欠额转余额”页签。
2. 打开目标记录，先运行只读预检查；预检查不会更改余额、流水、订单、库存、`disposition` 或 `confirmed_at`。
3. 核对订单、支付会话、唯一链上证据、确认数、截止时间、冻结汇率、预计入账金额与结算后余额。
4. 填写 1–500 字处理原因，完整输入订单号，勾选不可撤销确认，再通过二次确认弹窗。
5. 请求期间不得重复点击。同一 request ID 重试以及 `already_settled` 都按幂等成功处理，不得重复增加余额或释放库存。
6. 成功后核对用户余额流水、欠额 disposition、订单取消状态、支付会话关闭状态和库存释放状态。

安全边界：

- 操作员身份只能来自服务器认证会话；浏览器不能指定 `operator_user_id` 或 `settlement_source`。
- GET 和默认 POST 都必须保持只读；真实结算必须显式声明 `action=settle` 与 `dry_run=false`。
- service-role key 只存在于服务器，不得出现在浏览器响应、日志或文档。
- Migration 执行和业务结算是两项独立授权。Migration 成功不代表任何订单已经结算。
- `20260730_bep20_underpayment_confirmation_state.sql` 已有执行记录，不得修改或重复执行。

## 进入人工复核的情况

- 到账金额超过应付金额。
- 支付单过期后到账。
- RPC 查询失败或不稳定。
- 链上记录和订单状态冲突。
- 管理员认为交易需要人工复核。

## 后台可查看字段

后台支付详情中，USDT-BEP20 记录会展示：

- 支付方式
- 网络
- Chain ID
- 订单原币种
- 订单原金额
- 汇率
- 汇率来源
- 应付 USDT
- 实际到账 USDT
- 收款地址
- 付款地址
- TxHash
- 区块号
- 确认数
- USDT 合约地址
- 支付会话状态
- 异常原因
- 创建时间
- 确认时间
- 过期时间
- BscScan 交易链接

后台不展示 RPC URL、环境变量、私钥或助记词。

## 重新核验

管理员可以在支付详情中点击“重新核验”。

要求：

- 必须填写原因。
- 只能重新查询已提交 TxHash 的链上支付单。
- 已支付记录不会重复调用 `completePayment()`。
- 结果写入管理员审计日志。
- 不提供“直接改为已支付”的按钮。

## 禁止操作

- 不得仅凭用户截图把订单标记为已支付。
- 不得绕过 Transfer 日志校验。
- 不得根据钱包余额变化判断某个订单已到账。
- 不得让同一 Transfer 日志绑定多个订单。
- 不得在异常未核清时重复发货。

## 真实链上测试步骤

1. 配置真实 `BSC_RPC_URL`、`BSC_USDT_CONTRACT`、`BSC_RECEIVE_ADDRESS`。
2. 配置 `CNY_USDT_FIXED_RATE` 或真实汇率 Provider。
3. 创建小额测试订单。
4. 选择 USDT-BEP20 支付。
5. 确认页面显示的 USDT 金额和收款地址。
6. 从外部钱包向固定地址转入准确 USDT-BEP20 金额。
7. 提交真实 TxHash。
8. 确认数不足时应显示 `confirming`。
9. 确认数满足后订单变为已支付。
10. 再次提交同一 TxHash 不得重复支付。
11. 用同一 TxHash 尝试支付其他订单必须被拒绝。
12. 检查 `chain_transactions` 只有唯一记录。
13. 检查数字商品只交付一次。

本任务不自动发起真实链上转账。
