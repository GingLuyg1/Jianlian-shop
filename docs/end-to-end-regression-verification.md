# Jianlian Shop 全站端到端回归验收报告

更新时间：2026-06-29

本次验收遵守约束：未启动 `npm run dev`，未修改依赖、`.env.local`、`.next`、Nginx、PM2 或服务器配置，未执行 Supabase SQL，未新增购物车功能，未自动部署。验收方式为代码链路静态检查、关键 SQL/RPC 审计、类型检查和生产构建验证。

## 前台商品浏览结果

- 首页、分类页、商品列表、商品详情统一通过商品详情/立即购买链路，没有发现购物车入口作为购买前置步骤。
- 商品卡片使用 `/products/[id]` 和 `/checkout?product=<id>` 路由，未发现无效购物车跳转。
- 商品详情页读取真实商品数据，状态、库存、价格与图片按当前数据库字段展示。
- 已下架、售罄状态的最终购买权限仍依赖服务端 `create_order_with_item` 校验，前端展示只作为提示。
- 图片失败占位需继续在真实浏览器和真实素材环境复测，本轮未启动开发服务。

## 用户注册登录结果

- 注册/登录使用 Supabase Auth，用户资料由 `profiles` 链路创建或补建。
- 未登录订单创建接口会返回 401，未登录立即购买需由页面跳转逻辑继续保持。
- 会话读取集中在 Supabase 客户端/服务端模块，未发现前端输出 access token 的日志。
- 重复邮箱、会话过期、登录失败的真实表现需要在 Supabase 环境中用测试账号复测。

## 立即购买结果

- 商品卡片和商品详情均使用“立即购买”，没有“加入购物车”按钮。
- `/api/orders` 创建订单前读取当前登录用户，并调用数据库 RPC `create_order_with_item`。
- 订单创建时前端传入的商品价格不会被信任，最终价格由 RPC 重新读取数据库商品或 SKU 价格。
- 发现并修复：最新 SKU migration 覆盖的 `create_order_with_item` 原本只检查库存，未在订单创建阶段扣减库存，存在并发超卖风险。
- 多 SKU 的最终服务端链路支持 `sku_id`，但前台是否完整传入真实数据库 SKU 仍需要在对应商品详情交互中浏览器复测。

## 订单创建结果

- `create_order_with_item` 负责生成订单、订单项和初始状态日志，并返回真实订单编号。
- 本次修复后，订单创建会对商品或 SKU 行加 `FOR UPDATE` 锁，并在同一事务内扣减对应库存。
- 单规格商品扣减 `products.stock`；SKU 商品扣减 `product_skus.stock`。
- 库存不足、商品下架、SKU 不可购买时会在 RPC 内中断，不会只留下前端本地假订单。
- 订单号仍由服务端生成，不使用数组下标或前端生成值。

## 订单生命周期结果

- 后台订单状态变更通过 `/api/admin/orders/[orderId]` 和 `admin_update_order_status`，不是普通前端直接改表。
- 支付成功链路通过支付回调服务更新支付状态，再调用数字发货服务。
- 自动发货 RPC 支持重复调用幂等返回，避免重复发货。
- 未支付取消、已取消/已关闭不可继续支付等规则需结合现有订单状态 RPC 在真实数据库中复测。

## 用户中心结果

- `/account/orders` 和 `/account/orders/[orderNo]` 使用当前登录用户查询订单。
- 订单详情读取 `order_items`、状态日志与交付记录；交付内容通过独立接口延迟读取。
- `/api/orders/[orderNo]/delivery` 要求登录，只允许订单所属用户读取，并设置 no-store 响应头。
- 订单列表不直接返回完整卡密，敏感交付内容由交付接口按权限返回。

## 管理员业务链路结果

- 商品新增/编辑此前已修复保存后 dirty 状态与自动关闭问题，本轮未再次改商品编辑 UI。
- 后台订单、支付、库存相关接口通过管理员鉴权工具进入，普通用户不能直接调用管理员接口。
- 数字库存管理页面与订单详情交付面板已存在后台入口；完整导入和重试发货需要真实管理员账号复测。
- 审计日志服务会过滤 token、secret、card、code、卡密、密码等敏感字段。

## 权限隔离结果

- 用户订单 API 基于 `auth.getUser()` 和用户 ID 查询，只返回自己的订单。
- 交付内容 API 要求登录并通过 RPC 限制订单归属。
- 管理员 API 使用 `requireAdminApi` 或服务端管理员上下文。
- 未发现浏览器端使用 service_role key；service_role 仅在服务端模块中读取。

## 异常与并发结果

- 本次修复了订单创建阶段库存并发风险：商品/SKU 读取加行锁，扣减使用 `stock >= quantity` 二次条件。
- 自动发货 RPC 使用 `FOR UPDATE SKIP LOCKED` 锁定数字库存，避免同一库存并发重复交付。
- 支付回调服务会对重复回调和异常记录安全日志，不输出完整密钥字段。
- 网络中断、请求超时、多个浏览器同时编辑、真实重复回调需要在线环境或测试库复测，本轮不模拟真实支付成功。

## 购物车残留清理结果

- 活跃前台未发现购物车入口、加入购物车按钮、购物车页面或购物车 API。
- 已从 `supabase/migrations/20260629_multi_sku_core.sql` 移除 `cart_items` 建表、索引、触发器、RLS 策略和授权。
- 保留的 `components/products/ProductCard.tsx` 文本只是说明性代码注释：不显示购物车按钮。
- 保留的 `ShoppingCart` 是后台 Dashboard 图标名称，不构成购物车业务入口。
- 未删除线上历史数据库表，符合“不删除现有数据”的约束。

## 发现的问题

1. 最新 SKU migration 中仍包含 `cart_items` 表和 RLS 策略，与当前“不做购物车”产品规则冲突。
2. 最新 SKU migration 重新定义 `create_order_with_item` 后，库存只校验不扣减，可能导致最后一件并发超卖。
3. 最新 SKU migration 重新定义 `deliver_digital_order` 时参数名为 `p_operator_type`，但服务端 `delivery-service` 调用的是 `p_trigger_source`，会导致 RPC 参数不匹配。

## 已修复的问题

1. 移除 SKU migration 中的购物车建表、索引、触发器、RLS 策略和授权。
2. 给订单创建 RPC 的商品/SKU 读取增加 `FOR UPDATE` 行锁。
3. 订单创建 RPC 在同一事务中扣减商品或 SKU 库存，库存不足时整体失败。
4. 将自动发货 RPC 参数名改为 `p_trigger_source`，与服务端调用保持一致。

## 仍存在的问题

- 由于本轮禁止启动 `npm run dev`，页面交互、图片占位、浏览器刷新会话、真实账号登录等未做浏览器现场验证。
- 真实支付供应商未接入，支付成功、重复支付回调、真实自动发货只能通过已有服务链路静态验收。
- 如生产库已执行过旧版 `20260629_multi_sku_core.sql`，需要手动执行补丁 SQL 清理或重新应用修订后的 migration；本轮未自动执行 SQL。
- 多 SKU 前台完整选择到 `sku_id` 的交互需在真实商品数据上复测。

## 需要执行的 migration

- `supabase/migrations/20260629_multi_sku_core.sql`

如果线上数据库已经执行过旧版 migration，需要管理员手动补充：移除/忽略购物车相关对象、更新 `create_order_with_item` 和 `deliver_digital_order` 函数。不要在生产库直接删除历史购物车表，除非确认无人使用且已备份。

## 真实支付接入前阻塞项

1. 支付回调签名、订单支付流水和渠道配置需要用真实供应商沙箱验签。
2. 支付成功后订单状态、支付状态、自动发货和推广佣金需要在测试库跑真实闭环。
3. 支付失败、重复回调、超时回调和人工补单需要专门验收。

## 上线前阻塞项

1. 必须在测试库执行最新 migration，并验证订单创建、库存扣减、自动发货 RPC。
2. 必须用普通用户和管理员账号各跑一遍：注册、登录、购买、支付模拟/人工确认、发货、查看交付内容。
3. 必须验证生产环境变量、RLS 策略、域名回调地址和静态资源访问。
4. 必须确认未把 `.env.local`、`.next`、`node_modules` 和任何密钥提交到仓库。
