# 业务补偿处理手册

## 适用场景

补偿任务只用于数据库事务无法覆盖的外部事实或跨系统异常，例如：

- Provider 已确认支付成功，但站内订单状态更新失败。
- 外部退款已真实完成，但站内退款状态或余额流水写入失败。
- 数字交付已完成，但站内通知失败且需要人工确认。
- 缓存失效、邮件通知等非资金失败需要记录，但通常不属于资金补偿。

## 不适用场景

以下情况不应创建补偿任务：

- 数据库事务可直接回滚的普通失败。
- 表单校验失败。
- 用户权限不足。
- Provider 未配置导致无法发起支付或退款。
- 邮件发送失败但业务状态已明确，除非需要人工联系用户。

## 处理步骤

1. 进入后台：

```text
/admin/system/compensations
```

2. 按业务类型、状态或 Request ID 定位任务。
3. 打开关联订单、支付、退款或用户记录核对真实状态。
4. 如果仍需人工处理，标记为“人工审核”。
5. 完成核对并由正确业务入口修复后，标记为“已解决”。
6. 如果确认无需处理，标记为“已取消”。

每次处理必须填写原因，并写入 `admin_audit_logs`。

## 安全原则

- 不在补偿页面直接修改余额。
- 不在补偿页面直接修改支付成功状态。
- 不恢复已交付数字库存为可用。
- 不展示完整支付回调、密钥、Token 或卡密。
- 高风险修复必须使用对应业务 RPC 或服务端接口。

## 重试策略

当前版本不自动执行补偿重试。后续如果启用自动重试，需要满足：

- 操作具备幂等键。
- 重试不会重复入账、重复退款或重复交付。
- 超过最大次数进入 `manual_review`。
- 每次重试写入审计和 Request ID。

## 常见处理

| 场景 | 处理方式 |
| --- | --- |
| 支付成功但订单未更新 | 核对 Provider 和 `payment_sessions`，通过支付状态同步服务重放幂等处理。 |
| 充值成功但余额未入账 | 使用充值入账 RPC，确认未存在同业务流水后处理。 |
| 退款成功但状态未更新 | 使用退款后台入口完成状态和流水修复。 |
| 自动发货失败 | 在订单详情中使用幂等重试发货。 |
| 通知失败 | 进入邮件通知或站内通知重试，不修改资金状态。 |
