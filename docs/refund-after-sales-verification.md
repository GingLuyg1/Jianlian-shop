# 退款与售后联调验收

## 退款数据结构

新增 `refund_requests` 保存退款申请、审核和完成状态；新增 `refund_status_logs` 保存退款状态流水；复用 `balance_transactions` 保存余额退款资金流水；新增 `site_notifications` 保存站内通知。订单表兼容增加 `refund_status` 和 `refunded_amount`，用于展示未退款、处理中、部分退款、全额退款和失败状态。

## 退款状态机

状态值：`requested`、`reviewing`、`approved`、`processing`、`succeeded`、`rejected`、`failed`、`cancelled`。

用户只能创建 `requested`。管理员可将退款标记为处理中、拒绝、取消、失败、余额退款成功或外部渠道人工完成。`succeeded`、`rejected`、`cancelled` 为终态，重复处理返回幂等结果。

## 用户申请入口

订单详情页增加“退款 / 售后”板块。已支付且未取消、未全额退款、无处理中退款时显示“申请退款”。提交字段包括退款原因、说明、申请金额和联系方式。服务端会重新校验订单归属、支付状态、可退金额和重复申请。

## 可退款金额计算

通过数据库函数统一计算：订单实付金额 - 已成功退款金额 - 处理中退款金额。前端传入金额只作为申请金额，最终以服务端计算为准。失败和拒绝的退款不占用额度。

## 后台售后管理

新增 `/admin/refunds`，只允许超级管理员访问。列表展示退款单号、订单号、用户、金额、渠道、原因、交付状态、退款状态、申请时间和等待时长。详情抽屉支持余额退款、标记处理中、外部渠道人工完成、拒绝、取消和失败。

## 余额退款结果

余额退款通过 `admin_process_refund_request` 原子执行：锁定退款、锁定订单、校验可退金额、锁定用户余额、增加余额、写入 `balance_transactions`、更新退款和订单状态、写入通知。重复执行会复用已有 `business_type='refund'` 的完成流水，不重复入账。

## 外部退款适配结果

未接真实 Provider。外部渠道只能登记“处理中”或“人工退款完成”，完成时必须填写外部渠道参考号。系统不会显示自动退款成功，也不会调用不存在的真实退款接口。

## 订单与交付联动

全额退款后订单标记为 `refunded`，支付状态标记为 `refunded`。已交付数字库存不恢复销售；仅未交付且仍处于 `reserved` 的库存可释放为 `available`。退款成功不会触发自动发货。

## 库存释放规则

已交付库存保持 `delivered`，历史交付记录不删除。未交付且仅为订单预留的库存，退款成功后释放。部分退款不恢复已交付内容。

## 用户退款记录

新增 `/account/refunds` 展示当前用户自己的退款/售后记录。订单详情页也展示当前订单的退款记录。跨用户访问由 RLS 和 API 登录态共同限制。

## 站内通知结果

创建退款申请、退款状态更新时写入 `site_notifications`。通知不包含支付密钥、内部风控备注或完整数字交付内容。

## 发现的问题

当前项目原有订单系统只保存本地支付状态，真实 Provider 退款接口未接入，因此外部渠道退款只能做人工登记。

## 已修复的问题

- 缺少退款申请表和状态日志。
- 缺少可退款金额统一计算。
- 缺少用户退款申请入口。
- 缺少后台售后退款审核页面。
- 缺少余额退款原子入账和幂等保护。
- 缺少退款状态通知和审计链路。

## 仍存在的问题

- 需要在 Supabase 手动执行 `20260629_refund_after_sales.sql` 后功能才会真正可用。
- 未接真实支付宝、微信、币安或 USDT Provider 退款能力。
- 未实现外部通知，只生成站内通知。

## 需要执行的 migration

`supabase/migrations/20260629_refund_after_sales.sql`

## 真实 Provider 退款接入条件

- Provider 支持服务端退款接口。
- 退款请求可验签或可通过服务端查询确认。
- 保存真实 provider_refund_id。
- 回调幂等并可校验订单、金额和币种。
- Provider 成功确认后再将系统退款置为 `succeeded`。
