# 生产数据清理 Runbook

## 停止条件

- 没有完成数据库备份。
- 无法确认当前 Supabase 项目是目标环境。
- dry-run 结果包含真实订单、真实支付、真实余额流水或真实交付记录。
- 清理期间仍有用户下单、支付或库存导入。
- 操作者不能解释每一条候选记录的来源。

## 标准流程

1. 记录当前 Git commit、数据库项目、操作者和时间。
2. 暂停写入：维护公告、暂停支付入口、暂停库存导入和自动清理任务。
3. 完成 Supabase 备份，数字库存备份必须加密保存。
4. 在 SQL Editor 手动执行 `scripts/production-data-cleanup-dry-run.sql`。
5. 按表导出 dry-run 结果，逐条人工核对。
6. 对无法确认的数据标记“需人工确认”，不得删除。
7. 复制 `scripts/production-data-cleanup-template.sql` 到临时 SQL 草稿。
8. 逐段解除注释，只填入已批准 ID。
9. 每段执行前先 SELECT 影响数量。
10. 按子表到主表顺序处理。
11. 清理后重新执行 dry-run 和数据一致性巡检。
12. 恢复写入，观察订单、支付、库存和错误日志。

## 业务规则

- 真实订单不能删除。
- 余额流水不能随意删除。
- 审计日志原则上保留。
- 支付记录按财务和合规要求保留。
- 已交付或已预留数字库存不得物理删除。
- 测试访问统计可以按保留策略处理，但不能影响生产经营报表。

## 清理后校验

- `/admin/system/production-readiness` 无 P0 阻塞项。
- `/admin/system/database` 关键 migration 和表结构通过。
- 支付渠道未配置时保持关闭状态。
- 邮件配置状态清晰显示。
- 库存可售数量与数字库存一致。
- 订单创建、支付会话、自动发货和用户订单查询仍通过。
